# 가이드라인 2: 피드백 루프 시스템

## 핵심 원칙

> 독자 피드백과 최신 트렌드를 반영하되, 처음 정한 결말에서 벗어나면 안 됩니다.

---

## 루프 사이클

```
[집필] → [게시] → [반응 수집] → [분석] → [AI에게 전달] → [다음 화 집필]
  ↑_____________________________________________________________|
```

매 화를 이 사이클로 반복합니다. 중요한 것은 **분석 단계**입니다.
독자 반응을 그대로 AI에게 전달하면 오히려 일관성이 깨질 수 있습니다.

---

## 피드백 분류 방법

독자 반응을 받으면 먼저 세 가지로 분류하세요:

### A. 즉시 반영 가능
```
- 문체가 딱딱하다는 지적
- 특정 캐릭터의 대사가 어색하다는 피드백
- 오타, 표현 오류
```

### B. 검토 후 반영
```
- 특정 캐릭터를 더 자주 등장시켜 달라는 요청
- 전개 속도 조절 요청
→ 전체 마일스톤과 충돌하지 않으면 반영
```

### C. 반영 불가 (결말 방향과 충돌)
```
- 결말을 바꾸는 내용의 요청
- 핵심 마일스톤 이벤트를 없애달라는 요청
→ 반영하지 않되, 독자가 이해할 수 있는 다른 보상 제공
```

---

## 트렌드 주입 방법

### 분기별 트렌드 체크
3개월마다 자료조사/트렌드 폴더를 업데이트하고, 현재 인기 요소를 확인합니다.

### AI에게 트렌드 주입하는 방법
```
"현재 2026년 웹소설 시장에서 유행하는 [트렌드 키워드] 트렌드를
이 로그라인에 결합해줘. 단, 기존 결말과 세계관 규칙은 유지해야 해.
충돌 가능성이 있으면 먼저 알려줘."
```

### 트렌드 반영의 한계
- 트렌드는 **연출 방식**에 반영 (스탯창, 스킬 묘사, 대화 방식 등)
- **스토리 구조** 자체는 변경하지 않음

## AI 피드백 전달 프롬프트

```
[피드백 반영 지시]
_화에 대한 독자 반응이 나왔어.

[반영해야 할 내용]
- (A/B 유형 피드백)

[유지해야 할 내용]
- 최종 결말: (결말 재명시)
- 다음 마일스톤: (다음 고정 이벤트)

이 피드백을 반영해서 _화를 수정하거나 _화에 적용해줘.
결말 방향과 모순이 생기면 먼저 나에게 알려줘.
```
