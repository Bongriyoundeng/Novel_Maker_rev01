# 가이드라인 2: 스토리 바이블 구축 & 유지

## 스토리 바이블이란?

소설의 모든 **정설(Canon)** 이 담긴 데이터베이스입니다.
모든 에이전트는 집필 전 이 파일을 참조하고, 집필 후 이 파일을 업데이트합니다.

> **없으면 어떻게 되나?**
> AI는 초기 설정을 잊어버립니다. 5화에서 설정한 캐릭터 이름이 15화에서 바뀝니다.
> 독자는 이를 즉시 발견하고 신뢰를 잃습니다.

---

## 스토리 바이블 구성 요소

| 섹션 | 내용 | 중요도 |
|------|------|--------|
| 고정 결말 | 절대 변경 불가. 모든 에이전트의 나침반 | ★★★★★ |
| 마일스톤 | 반드시 일어나야 할 핵심 사건 | ★★★★★ |
| 세계관 규칙 | 마법/시스템/사회 법칙 | ★★★★★ |
| 캐릭터 레지스트리 | 현재 상태 추적 | ★★★★☆ |
| 플롯 비트 로그 | 집필 완료 기록 | ★★★★☆ |
| 복선 & 비밀 격리 | 독자에게 미공개 정보 | ★★★☆☆ |

## 고정값 vs 변동값 원칙

| 구분 | 내용 | 변경 가능 여부 |
|------|------|--------------|
| **고정값** | 최종 결말, 핵심 마일스톤, 세계관 핵심 규칙 | 불가 (사령관만 변경 결정) |
| **변동값** | 에피소드 세부 내용, 대화, 연출 방식, 부가 설정 | 가능 (에이전트 재량) |

## 스토리 바이블 작성 순서

### 프로젝트 시작 시 (1회)
```
1. story_bible.md 템플릿 복사 → 프로젝트 폴더 최상위에 배치
2. '고정 결말' 섹션 먼저 작성 — 가장 중요
3. '5단계 마일스톤' 작성 — 핵심 사건 확정
4. '세계관 규칙' 작성 — 세계관 제작자 에이전트 활용
5. '캐릭터 레지스트리' 초기 인물 등록
6. 'AI 에이전트 공유용 요약' 섹션 완성
```

### 화 완료 후 (매 화마다)
```
1. 세계관 제작자 에이전트에게 업데이트 요청
2. 플롯 비트 로그에 해당 화 기록
3. 변경된 캐릭터 상태 업데이트
4. 새로 확정된 사실 추가
5. 공개된 복선은 격리 섹션에서 이동
```

## 마이크로 요약과 델타 타일

장기 연재에서 가장 큰 문제는 **AI의 컨텍스트 윈도우 한계**입니다.
전체 원고를 매번 붙여넣을 수 없습니다.

### 해결책 1: 마이크로 요약
- 매 화 완료 후 2문장으로 핵심 결과를 요약
- 다음 화 집필 시 이전 화 마이크로 요약을 순차 입력
- 템플릿: `_template/micro_summary.md`

### 해결책 2: 델타 타일
- 최근 화에서 **변한 것만** 담은 간략한 업데이트
- 오래된 내용을 다시 설명하지 않아도 됨
- micro_summary.md의 '델타 타일' 섹션 사용

```
[좋은 예 - 델타 타일 사용]
"이전 내용은 생략. 6화 이후 변경 사항: 주인공이 길드에 가입함,
민준이 주인공의 정체를 의심하기 시작함."

[나쁜 예 - 전체 재설명]
"이 소설은 1화부터 시작해서 주인공이 회귀를 하고..."
```

## 복선 격리 전략

독자에게 너무 일찍 반전을 노출하는 것을 방지하는 메커니즘입니다.

```
[산문 스타일리스트에게 주는 지시]
story_bible.md의 '복선 & 비밀 격리' 섹션은 읽지 마세요.
그 섹션의 내용을 집필에 반영하거나 암시하지 마세요.
```

이를 통해 AI가 의도치 않게 독자에게 결말을 노출하는 것을 방지합니다.

## 흔한 실수와 해결법

| 실수 | 결과 | 해결법 |
|------|------|--------|
| 결말을 모호하게 설정 | AI가 열린 결말로 흘러감 | 결말을 구체적 장면 단위로 기술 |
| 화 완료 후 업데이트 미룸 | 누락 발생, 설붕 위험 증가 | 화 완료 즉시 업데이트 원칙 |
| 세계관 규칙을 AI에게 맡김 | 느슨한 규칙으로 설붕 발생 | 핵심 규칙은 사령관이 직접 확정 |
| 바이블 요약이 너무 길어짐 | 컨텍스트 낭비 | 요약은 핵심 3가지만 유지 |
